# AiLang Quick Reference (v0.3.0)

AiLang is a Rust-implemented compiled language that transpiles to C99 and links via `clang -O2`. The source syntax is optimized for low token count when generated by an LLM; the binary runs within ~8% of hand-written C.

## Mental model

- Default integer type is **`i64`**. Unannotated function parameters and locals are `i64` unless context says otherwise.
- All language keywords are **2 characters** (`fn`, `lp`, `mt`, `rt`, `el`, `mu`, `en`, …). Do not substitute `for`/`while`/`match`/`return`/`else`/`let`/`enum`.
- Top-level statements **auto-wrap into `fn main()`**. Don't write `fn main` yourself unless you specifically need to.
- The trailing expression of a function body is the **return value** — no `rt` needed at the end.
- Single-statement bodies skip `{...}`. `if x>0 print(x)` is legal.
- Trailing `if` / `mt` expressions are returned implicitly: `fn classify(n) { if n<0 "neg" el if n==0 "zero" el "pos" }` works with no `rt`.
- **Stdlib imports are auto-injected.** Just call `sqrt(x)`, `must_listen(...)`, `http_recv_request(fd)`, `redis_set(c, k, v)` etc. — the driver figures out which `std/*.ail` to pull in. Only write `im "std/X.ail"` when you want to be explicit (or for non-stdlib user modules). Unused modules are never parsed.

## Keywords (every one is exactly 2 chars)

| kw | meaning |
|----|---------|
| `fn` | function or lambda |
| `if` / `el` | if / else |
| `lp` | loop — unifies `for`-in **and** `while` |
| `rt` | return (only needed for early return) |
| `mt` | match |
| `st` | struct |
| `en` | enum / algebraic data type |
| `tr` | trait (parsed; runtime support minimal) |
| `im` | import |
| `ex` | extern (C ABI / FFI) |
| `mu` | mutable variable / mutable param marker |
| `br` / `ct` | break / continue |
| `as` | rename in `im` |
| `in` | iterator binding in `lp` |
| `ok` | construct an `!T` success value |
| `true` `false` `nil` | literals |

Type names (also keywords): `i8 i16 i32 i64 u8 u16 u32 u64 f32 f64 bool str bytes`.

`_` is the **wildcard pattern** (not a regular identifier).

## Operators

```
arithmetic  +  -  *  /  %         (+ also concatenates when an operand is str)
compound    +=  -=  *=  /=  %=
compare     ==  !=  <  <=  >  >=
logical     &&  ||  !
bitwise     &  |  ^  ~  <<  >>
string      ++                    (explicit concat — legacy; prefer +)
range       ..  ..=               (exclusive / inclusive)
declare     :=                    (NEW binding — use this to introduce a name)
assign      =                     (re-assign an existing mu binding)
return ty   ->
match arm   =>
pipe        |>                    (x |> f(a)  becomes  f(x, a))
nil coal    ??                    (a ?? b: a if non-nil else b)
ternary     cond ? a : b
optional    ?T                    (type prefix: nil allowed)
result      !T                    (type prefix: fallible — see Error handling)
try         expr?                 (postfix: propagate err to enclosing !T fn)
```

**Critical:** `:=` introduces a new name. `=` mutates an existing `mu`-binding. Mixing them up is the single most common mistake.

## Bindings

```
x := 10           // immutable binding
mu n := 5         // mutable — required to write `n = ...` later
n = 7             // ok because n is mu
n += 1            // compound assign also requires mu
```

## Functions and closures

Named function:
```
fn add(a, b) a + b                            // expression body
fn fib(n) {                                    // block body
  if n < 2 rt n                                // braceless if + early return
  fib(n-1) + fib(n-2)                          // trailing expr = return value
}
fn greet(name:str) "Hello, " + name + "!"      // annotate non-i64 params
```

Lambda (closure literal) — exact syntax is `fn(params) body`:
```
double  := fn(x) x * 2                         // closes over nothing
add     := fn(a, b) a + b
greeter := fn(name:str) "Hello, " + name + "!"

threshold := 3                                  // captured by value
gt := fn(x) x > threshold                       // gt(5) → true

println(double(7))                              // 14
```

Higher-order: pass lambdas to `map` / `filter` / `reduce` (provided by stdlib or the user):
```
nums := [1, 2, 3, 4, 5]
doubled := map(nums, fn(x) x * 2)               // [2,4,6,8,10]
odds    := filter(nums, fn(x) x % 2 == 1)
total   := reduce(nums, 0, fn(a, b) a + b)
```

**Closure gotcha — capture is by value at creation time.** Later mutation of the captured `mu` is invisible to the lambda. If you need the latest value, pass it as a parameter.

**Lambda syntax gotchas:** it is `fn(x) body`, NOT `|x| body` (Rust) or `(x) => body` (JS) or `lambda x: body` (Python).

Mutable parameter — receive a value, modify the local copy, return it:
```
fn shift(mu n, by) { n = n + by; n }
println(shift(10, 3))                           // 13
```

Inference rules (unchanged):
- Unannotated parameter → `i64`
- Return type inferred from the tail expression: `str` if it produces a string, `void` for a call to a void fn like `println`, else `i64`. For lambdas, the same rules apply.
- Annotate `:str` / `:bool` / `:[T]` / `:{K:V}` / `:Shape` (custom type) explicitly when the param isn't an i64.

## Control flow

```
// for-in (exclusive range; ..= is inclusive)
lp i in 1..10 {
  print(i); print(" ")
}

// while form — same lp keyword
mu n := 5
lp n > 0 {
  println(n)
  n -= 1
}

// if / el
if x > 0 {
  println("positive")
} el if x == 0 {
  println("zero")
} el {
  println("negative")
}

// match — arm separator is `;`
mt (i%3, i%5) {
  (0,0) => println("FizzBuzz");
  (0,_) => println("Fizz");
  (_,0) => println("Buzz");
  _     => println(i);
}

// map iteration with tuple destructure
lp (k, v) in m {
  print(k); print(":"); println(v)
}
```

## Algebraic data types (`en`)

Declare with `en Name { Variant, Variant(field:Type, ...), ... }`. Variants can be unit-only or carry typed payloads. Field type annotations are **required** in the declaration. Construct with bare `Variant` or `Variant(args)` — no namespace prefix. Destructure in `mt`:

```
en Shape {
  Circle(r:i64),
  Square(side:i64),
  Rect(w:i64, h:i64),
}

fn area(s:Shape) {
  mt s {
    Circle(r)  => r * r * 3;          // π ≈ 3
    Square(s)  => s * s;
    Rect(w, h) => w * h;
  }
}
println(area(Circle(5)))              // 75

en Color { Red, Green, Blue }         // classic enum (unit variants)
en Maybe { None, Some(v:i64) }        // Maybe-like
en Expr { Num(v:i64), Add(l:Expr, r:Expr) }  // recursive — self-ref payloads are heap-boxed
en Tree { Leaf(v:i64), Branch(kids:[Tree]) } // recursion through an array child-list
```

**Recursive ADTs work** — a variant may carry its own enum, directly
(`Add(l:Expr, r:Expr)`), through an array (`Branch(kids:[Tree])`), and
**mutually** across two enums (`en Expr`↔`en Stmt`). This is how you build an AST.

A **non-exhaustive `mt` on an enum** warns (naming the missing variants) but still
compiles — cover every variant or add a `_` catch-all.

**Gotchas:**
- Variant constructors are **bare**, not namespaced: `Some(42)` and `None`, NOT `Maybe::Some(42)` or `Maybe.Some(42)`.
- Patterns bind **positionally**: `Circle(r)` works regardless of the field name in the decl. No `..` rest, no guards.
- Two enums in the same module **cannot share** a variant name (single global variant table).

## Generics

Type parameters in `<...>` after the fn name. The body is checked once; codegen monomorphizes one C function per concrete call-site type:

```
fn id<T>(x:T) -> T x
println(id(42))                       // 42    (i64 specialization)
println(id("hello"))                  // hello (str specialization)

fn first<T>(arr:[T]) -> T arr[0]
println(first([10, 20, 30]))          // 10
println(first(["alice", "bob"]))      // alice

fn pair<A, B>(a:A, b:B) -> A a        // multiple type params
```

`T` can appear bare, as `[T]`, or as `{K:V}`, and may be a primitive **or a user struct/enum** (`id(Point(3, 4))`, `fst(Circle(5), Square(4))`). Generics no longer collapse to `i64` — each call-site type gets its own specialization. (T = fn-pointer is still future work.)

## Error handling: `!T` + `?` propagation

`!T` is a fallible result type. Constructors:
- `ok(v)` — polymorphic success
- `err(msg)` — polymorphic error; the T comes from the enclosing fn's `-> !T` return type. Outside a `-> !T` fn this is a compile error (since T can't be inferred).
- `err_i64(msg)`, `err_str(msg)`, `err_bool(msg)`, `err_f64(msg)` — type-suffixed forms; only needed when you're constructing a result outside its owning fn, which is rare. Prefer plain `err(msg)`.
- Inspectors: `unwrap(r)`, `is_ok(r)`, `is_err(r)`, `err_msg(r)`

Postfix `?` propagates the first error to the enclosing fn's `!T` return:

```
fn parse_int_safe(s:str) -> !i64 {
  if regex_match("^-?[0-9]+$", s) rt ok(str_to_int(s))
  err("not a number: " + s)         // T inferred as i64 from the fn's `-> !i64`
}

fn parse_sum(a:str, b:str, c:str) -> !i64 {
  x := parse_int_safe(a)?              // propagates on err
  y := parse_int_safe(b)?
  z := parse_int_safe(c)?
  ok(x + y + z)
}

r := parse_sum("1", "2", "3")
if is_ok(r) println(unwrap(r)) el println(err_msg(r))
```

**Gotchas:**
- Use plain `err("msg")` inside a `-> !T` fn — the T is inferred from the fn signature. The type-suffixed `err_i64`/`err_str`/`err_bool`/`err_f64` exist for the rare case where you're constructing a result outside its owning fn (typically not needed).
- `unwrap` aborts on error. Use `is_ok(r)` first if you need to handle the error path.
- `?` only works inside a fn whose return type is `!T` for the same `T`. **Implicit `main` is not `!T`-typed** — wrap fallible code in `fn run() -> !i64 { … }` and call it from main.

## Structs (`st`)

```
st Point { x:i64; y:i64 }              // fields separated by ; or ,

p := Point(3, 4)                        // positional ctor — args go to fields in decl order
q := Point { x: 3, y: 4 }               // named ctor — equivalent, more verbose
println(p.x)                            // 3
println(p)                              // Point { x: 3, y: 4 } (auto pretty-print)
```

**Prefer `Point(3, 4)`** — it's the same number of bytes shorter as bare enum variant construction and reads naturally when field order is obvious. Fall back to the named form when there are more than ~3 fields or the ordering would be ambiguous.

## Arrays, maps, bytes

```
nums  := [10, 20, 30]                   // [i64]
words := ["alice", "bob"]               // [str]
println(len(nums))                      // 3
println(nums[2])                        // 30
lp x in nums { total += x }

// Maps — multiple instantiations now supported
m1 := {1: 10, 2: 20}                    // {i64:i64}
m2 := {"alice": 90, "bob": 75}          // {str:i64}
m3 := {"name": "Ada", "lang": "AiL"}    // {str:str}
m1[4] = 40                              // insert / set
println(has(m1, 4))                     // true
println(m1[99])                          // missing key → 0 (no exception)

// bytes — distinct from str, may contain \0
b := str_to_bytes("hello\0world")
println(len(b))                          // 11
println(b[0])                            // 104   (= 'h')
slice := bytes_slice(b, 6, 11)           // bytes "world"
println(bytes_to_str(slice))             // world
```

Supported instantiations today: `[i64]`, `[str]`, `[bytes]`, **`[Struct]` / `[Enum]`** (arrays of user types, including self-recursive ones like `[Tree]`), `{i64:i64}`, `{str:i64}`, `{str:str}`, and **`{str:Struct}` / `{str:Enum}`** (symbol tables — `has`/`keys`/`values`/`(k,v)` iteration all work). Non-`str` map keys with aggregate values aren't supported yet. Other combos may or may not be implemented — check examples before assuming.

**Empty `{}` / `[]` infer `(K,V)` / element type from later usage.** Prefer the short form when the binding is filled in by a loop or a sequence of writes — the type annotation is redundant:

```
counts := {}                    // type inferred as {str:i64} from below
lp w in words { counts[w] += 1 }

squares := {}                   // inferred as {i64:i64}
lp n in 1..10 { squares[n] = n * n }
```

Only annotate `mu m:{K:V} := {}` when the binding has no later writes the compiler can read (rare).

## Pipe operator and string concat

```
fn double(x) x * 2
fn square(x) x * x

println(5 |> double |> square)           // square(double(5)) = 100

msg := "Hello, " + "AiLang" + "!"        // + concatenates when an operand is str
println(len(msg))                        // 14
```

## String interpolation

Use `${expr}` inside a regular `"..."` string. **Any expression** works — bare idents, field access, calls, indexing, arithmetic — and the value is stringified automatically (`i64` / `f64` / `bool` / `str` all work):

```
name := "alice"
age  := 30
nums := [10, 20, 30]

println("hello, ${name}")                        // hello, alice
println("${name} is ${age}")                     // alice is 30
println("next ${age + 1}, up ${to_upper(name)}") // next 31, up ALICE
println("nums[0]=${nums[0]}, n=${len(nums)}")    // nums[0]=10, n=3
```

**Prefer interpolation over `format(...)`** for building strings — fewer tokens (no `%lld`/`%s` placeholders, no trailing arg list) and easier to read. Reach for `format` only when you need real printf control (width, precision, zero-padding).

**Rules:**
- Prefix is `${` (dollar **and** brace). A `$` followed by anything else is literal: `"cost $5"` and `"$dollar"` are not interpolations.
- JSON-shaped strings are unaffected: `"{\"name\":\"bob\"}"` has no `${` so it stays verbatim.
- The text inside `${...}` is re-parsed as a full expression. Anything that isn't one valid expression (e.g. `${not an ident here}`) stays as literal text.
- Brace matching is naive — the first `}` closes the `${`. An expression that itself contains `}` (a map/block literal) would truncate; store it in a local first. `${p.x}`, `${f(a)}`, `${m[k]}` are all fine.
- The expression **can't contain a `"`** — the outer string's quote closes early. A string-literal map key like `${m["the"]}` won't parse; hoist it: `t := m["the"]; "...${t}..."`. A variable key `${m[k]}` is fine.
- Desugars to `"prefix" + to_str(expr) + "suffix"`, interchangeable with `+` concat.

## FFI: C functions, headers, libraries

```
ex fn puts(s:str) -> i32                 // libc — declared, linked directly
ex fn abs(n:i32) -> i32

puts("hello via libc puts!")
println(abs(-7))                         // 7
```

`ex fn` declares an extern with the C ABI — no body. libc and libm are always linked.

**Pull in a C header with `cinc`, link a library with `ex "lib"`:**

```
cinc "zlib.h"                            // → #include <zlib.h> in the generated C
ex "z" fn zlibVersion() -> str           // declares the fn AND links -lz
println(zlibVersion())                   // 1.2.12
```

- `cinc "h.h"` emits `#include <h.h>`, so the header's macros, typedefs and struct definitions are visible to the generated C.
- `ex "lib" fn …` both declares the symbol (so sema accepts the call) and tells the driver to link `-llib`. Bare `ex fn` (or `ex "c" fn`) = libc, no extra link.
- A header typedef can be named directly in a signature: `ex "sqlite3" fn sqlite3_open(path:str, db:*sqlite3) -> i32` — unknown type names are emitted to C verbatim, and `cinc "sqlite3.h"` supplies the real definition. Use `*T` for pointers, `&x` to take an address.

**Structs work, by value and by pointer:**

```
cinc "stdlib.h"
ex fn div(a:i32, b:i32) -> div_t       // returns a struct BY VALUE
r := div(17, 5)
println("${r.quot} ${r.rem}")          // 3 2 — read fields with `.`

ex fn fopen(p:str, m:str) -> *FILE     // opaque struct POINTER, pass it around
ex fn fclose(f:*FILE) -> i32
```

- **By-value struct return + `.field` read** works for typedef'd structs (`div_t`, `ldiv_t`, …).
- **Opaque struct pointers** (`*FILE`, `*sqlite3`) pass through cleanly — declare with `*Typename`.
- **`.field` through a pointer auto-selects `->`.** `&x` yields a real `*T`, and `p.field` on any `*Struct` codegens as `p->field`. This also gives **pointer out-params for your own `st` structs** — `fn bump(c:*Counter) { c.value = c.value + 1 }` then `bump(&ctr)` mutates the caller's struct. See [`examples/ptr_fields.ail`](../../examples/ptr_fields.ail).

**Limits (current FFI):** you still hand-write each `ex fn` signature — no header auto-binding/bindgen. **Tag-only structs can't be named** — `struct tm`/`struct stat` (no typedef) have no AiLang spelling, since a bare name emits `tm` not `struct tm`; wrap field access in a tiny C function, or use a typedef'd alias header. `#define` macros and constants aren't visible as AiLang names (sema would see them undefined); wrap one in a tiny C function if you need it. Only system include/lib search paths are wired. If your `ex fn` signature's types don't match a prototype the header already declares (e.g. `u64` vs zlib's `uLong`), clang reports `conflicting types` — name the header's own typedef in the signature to match exactly.

## Modules and stdlib

```
im "mathlib.ail"                         // user module — relative path
im "std/math.ail"                        // stdlib
```

Resolution: first relative to the importing file, then to cwd. Imported `fn main` is silently dropped. Available `std/*.ail` modules:

| module | what it provides |
|--------|------------------|
| `std/math.ail`  | libm wrappers (`sqrt`, `pow`, `sin`, `cos`, `tan`, `log`, `log2`, `log10`, `exp`, `floor`, `ceil`), plus `min`, `max`, `ipow` (integer power), `gcd`, libc `abs`/`rand`/`srand` |
| `std/str.ail`   | `eq(a, b)`, `parse_int(s)`, libc `strcmp`/`atoi` |
| `std/time.ail`  | `now_ms()`, `now_us()`, `mono_ms()`, `time_iso(ms)`, `now_iso()`, `tick()`/`since(t)` stopwatch, `sleep_ms`/`sleep_s` |
| `std/sock.ail`  | `tcp_listen(host, port)`, `tcp_accept(fd)`, `tcp_connect(host, port)`, `sock_send`, `sock_send_str_all(fd, s)`, `sock_recv(fd, max) -> bytes`, `sock_close`. Returns `-1` on error. Convenience helpers for server boilerplate: `env_int(name, default)` (parse env var as i64), `must_listen(host, port, banner)` (listen + log + `exit(1)` on failure; pass `""` as banner to skip the log line). |
| `std/http.ail`  | `http_text(status, body)`, `http_json(status, body)`, request parsers `http_method/http_path/http_header(req, name)/http_body/http_content_length`, `http_recv_request(fd)` (handles Content-Length). Keep-alive variants suffixed `_ka`. |
| `std/redis.ail` | `redis_connect(host, port)`, `redis_close`, `redis_ping`, `redis_set(fd, k, v)`, `redis_get(fd, k)`, `redis_incr(fd, k)`, `redis_del(fd, k)`, `redis_exists(fd, k)`. Returns `""` on nil/error. |
| `std/pg.ail`    | Postgres via libpq. `pg_connect(dsn)`, `pg_status`, `pg_exec(conn, sql) -> res`, `pg_ok(res)`, `pg_value(res, row, col)`, `pg_nrows`, `pg_ncols`, `pg_escape(conn, s)`, `pg_clear`, `pg_close`. Helpers: `pg_must_connect`, `pg_one(conn, sql)` (first cell as str), `pg_first_col(conn, sql) -> [str]`, `pg_print_table(res)`. **Always `pg_escape` user input.** |
| `std/tls.ail`   | OpenSSL TLS. `tls_server_ctx(cert, key)`, `tls_client_ctx()`, `tls_accept(ctx, fd) -> ssl`, `tls_connect_fd(ctx, fd)`, `tls_send_str_all(ssl, s)`, `tls_recv(ssl, max) -> bytes`, `tls_close`, `tls_free_ctx`, `tls_error()` |
| `std/ws.ail`    | WebSocket server-side. `ws_handshake_response(client_key)` (101 Switching Protocols string), `ws_send_text(fd, payload)`, `ws_recv_text(fd) -> str`. Hard limit: payload < 126 bytes (short frames only). |
| `std/json.ail`  | Flat (one-level) JSON parsers: `parse_flat_obj_str(s) -> {str:str}`, `parse_flat_obj_int(s) -> {str:i64}`. Result map includes a sentinel `"_"` key. |

### Containers (arrays / maps)
| Name | Signature | Notes |

| `keys(m)`       | `({K:V}) -> [K]`     | order undefined |
| `values(m)`     | `({K:V}) -> [V]`     | order undefined |

## Compile and run

```bash
ailangc run prog.ail                # compile + execute (forwards exit code)
ailangc compile prog.ail            # produces ./prog binary
ailangc compile prog.ail -o out     # custom output path
ailangc compile --emit-c prog.ail   # print generated C to stdout, skip clang
ailangc compile --keep-c prog.ail   # keep the generated prog.c next to the binary
ailangc parse  prog.ail             # dump AST
ailangc tokens prog.ail             # dump token stream
```

The output binary lands next to the source. The generated `.c` (or `.ll`)
is a temporary build artifact: it's deleted once the binary links, unless
you pass `--keep-c`. A *failed* compile always leaves it behind so the
codegen can be inspected.

## What is NOT supported (don't generate these)

- `?` postfix at top level (implicit-main scope) — wrap in `fn run() -> !T { … }`. (`!T` and `?` now work for T = primitive **or** struct/enum.)
- `match` guards (`if cond` on a pattern arm), rest patterns (`..`), or `@` bindings
- Multi-statement `mt` arm bodies — an arm body is one expression; wrap multiple statements in a `{...}` block expression. (`if`/`mt`/blocks otherwise work as expressions: `x := if c {a} el {b}`, `x := mt v {…}`.)
- Closures that mutate captured variables (capture is by-value snapshot)
- Interpolation where the expression itself contains a literal `}` (e.g. a `${ {1:2} }` map/block literal) — the first `}` closes the `${`. `${p.x}`, `${f(a)}`, `${m[k]}`, `${a+b}` are all fine; only hoist the rare `}`-containing case into a local.
- Python-style `f"..."` or `{x}` (no `$`) interpolation — AiLang only recognises `${x}`
- `tr` trait method dispatch (the keyword parses but resolution is minimal)
- Long WebSocket frames (≥126 bytes) in `std/ws.ail` — current impl is short-text only
- Nested JSON via `std/json.ail` — only one-level objects

## Idiomatic vs non-idiomatic

Prefer:
- Implicit `main` for top-level scripts
- **Skip stdlib `im` lines** — call `sqrt(x)`, `must_listen(...)`, `redis_set(...)` directly; the driver injects the right `std/*.ail`
- Expression-bodied fns: `fn add(a, b) a + b`
- Trailing-expression return (no `rt` at end of body)
- Trailing `if` / `mt` as return value
- Braceless single-statement bodies
- `+` for string concat (not `++`)
- **`"...${expr}..."` interpolation** over `format(...)` for plain string building
- `lp x in coll` over manual index counters
- `mt` over chained `if`/`el if` for value matching
- Bare variant names: `Some(42)`, NOT `Maybe::Some(42)`
- **Positional struct ctor**: `Point(3, 4)` over `Point { x: 3, y: 4 }`
- `fn(x) body` lambdas, NOT `|x| body` (Rust) or `(x) => body` (JS)
- **Polymorphic helpers**: `err(msg)` over `err_i64`, `abs(x)` over `abs_i64`/`abs_f64`, `to_str(x)` over `int_to_str`/`float_to_str`/`bytes_to_str`

Avoid:
- `let`/`var`/`enum`/`return`/`for`/`while`/`else` — wrong keywords
- `: i64` and `-> i64` annotations when default already gives i64
- `Err("msg")` / `Result::Ok` — it's `ok(v)` and `err("msg")`
- `Maybe.Some(v)` — variants are global, bare
- Type-suffix conversion clutter (`int_to_str`, `bytes_to_str`, `err_i64`, `abs_i64`) when the polymorphic form works
- `format("...%lld...", x)` when plain interpolation `"...${x}..."` would do

## Worked examples

### Hello world (implicit main)
```
println("hello, AiLang")
```

### FizzBuzz with tuple match
```
lp i in 1..16 {
  mt (i%3, i%5) {
    (0,0) => println("FizzBuzz");
    (0,_) => println("Fizz");
    (_,0) => println("Buzz");
    _     => println(i);
  }
}
```

### Recursive fib + iterative fib
```
fn fib(n) {
  if n < 2 rt n
  fib(n-1) + fib(n-2)
}
println(fib(30))                          // 832040

fn fib_iter(n) {
  if n < 2 rt n
  mu a := 0; mu b := 1; mu i := 2
  lp i <= n {
    t := a + b
    a = b; b = t
    i += 1
  }
  b
}
println(fib_iter(20))                     // 6765
```

### ADT + match (shapes)
```
en Shape {
  Circle(r:i64),
  Square(side:i64),
  Rect(w:i64, h:i64),
}

fn area(s:Shape) {
  mt s {
    Circle(r)  => r * r * 3;
    Square(s)  => s * s;
    Rect(w, h) => w * h;
  }
}

shapes := [Circle(5), Square(4), Rect(3, 6)]
lp s in shapes { println(area(s)) }
```

### Higher-order with closures
```
fn map_i64(arr:[i64], f) {
  mu out := [0]
  out[0] = f(arr[0])
  lp i in 1..len(arr) { out[i] = f(arr[i]) }   // sketch — real impl appends
  out
}

nums := [1, 2, 3, 4, 5]
threshold := 2
big := filter(nums, fn(x) x > threshold)       // [3,4,5]
sum := reduce(nums, 0, fn(a, b) a + b)         // 15
println(sum)
```

### Generics — id and first
```

println(id("hello"))                           // hello

```

### Result / error propagation
```
im "std/str.ail"

fn parse_pair(a:str, b:str) -> !i64 {
  x := parse_int(a)?                            // propagates err
  y := parse_int(b)?
  ok(x + y)
}

fn run() -> !i64 {
  total := parse_pair("12", "30")?
  ok(total)
}

r := run()
if is_ok(r) println(unwrap(r)) el println(err_msg(r))
```

### Bubble sort (arrays + nested loop + swap)
```
mu arr := [5, 2, 8, 1, 9, 3, 7, 4, 6]
n := len(arr)
lp i in 0..n {
  lp j in 0..(n-i-1) {
    if arr[j] > arr[j+1] {
      t := arr[j]
      arr[j] = arr[j+1]
      arr[j+1] = t
    }
  }
}
lp x in arr { print(x); print(" ") }
println("")
```

words := ["alice", "bob", "alice", "carol", "bob", "alice"]

## Top gotchas an LLM coming from Rust/Python/Go gets wrong

1. **`err(msg)` works — but only inside a `-> !T` fn.** Sema reads T from the enclosing signature. Outside that context it's a compile error; use `err_i64`/`err_str`/etc. only if you really need to construct a result outside its owning fn. `ok(v)` is also polymorphic.
2. **`?` only works inside `-> !T` fns.** Implicit `main` is not `!T`-typed. Wrap in `fn run() -> !i64 { … }`.
3. **Lambda syntax is `fn(x) body`** — NOT `|x| body`, `(x) => body`, or `lambda x: body`. Capture is by value at creation; later mutation of the captured `mu` is invisible to the lambda.
4. **`mt` arm separator is `;`** (not `,`). Variant patterns bind positionally — `Circle(r)` doesn't have to match the field name in the decl.
5. **Variant constructors are bare**, not namespaced: `Some(42)` / `None`, not `Maybe::Some(42)` or `Maybe.Some(42)`. Two enums in the same module can't share a variant name.
6. **Real types in keywords**: it's `en`, not `enum`; `st`, not `struct`; `fn`, not `function`; `mt`, not `match`; `rt`, not `return`.
7. **String interpolation needs `$`**: `"hi ${name}"` works, `"hi {name}"` does NOT — the bare-brace form (Python f-string style) stays as literal text. Inside `${...}` any expression works (`${obj.field}`, `${f(x)}`, `${a+b}`); only a literal `}` within the expression truncates it.

When in doubt, mimic the shape of programs in `examples/*.ail` rather than translating from another language.
